## CI

持续集成/持续发布，由CI服务器来管理

CI服务器的输入就是不同分支的代码

在进行一系列的测试，构建之后，将准确的产品进行部署发布



CI开启一个过程，可以由
- 代码提交，就开始触发，webhook
- 内部设定周期，主动从代码服务器拉取代码




每次CI启动一个pipeline，对于同一个项目，不同的分支/相同的分支（前后两次提交，可能会有后提交的代码，先构建完成的情况），
pipeline应该是并行的，多个构建没有强依赖与相互影响
- 使用构建编号作为全局变量，标识符


## pipeline

### phase

一个pipeline由多个phase组成

每个pipeline的运行状态都要被追踪

每个phase应该返回成功/失败，不管什么结果，
都要有所反馈

一个phase失败，则pipeline失败

可以通过邮件/IM之类的方式，传递信息

todo： 研究jenkins pipeline，对pipeline有更深入的了解

### test

一般来讲，test是第一个phase，

错误分为很多种

- 语法错误，无法执行的代码
- 构建错误，由语法错误导致
- 逻辑错误，代码内部的逻辑出错，得不到预期的结果
- 实现错误，一切都很完美，但是做出来的东西不是当初所设计的东西

单元测试
- 只能测试出前3种情况
- 大范围的代码覆盖率
- 可以避免很多问题

这里的测试应该是自动化的，可以帮忙检测出很多低级的问题


### build

因为每一次都在持续的发布，尽可能的优化，将每一次构建的时间缩短，
效率提高

- 使用缓存
- 优化代码
- 使用其它服务器辅助构建

### deploy

虽然概率很小，但是发布过程出错也是有可能的
旧的服务已经关闭，新的服务还没有启动，整个pipeline失败退出



## 环境

分为3个环境
- 开发环境，是内部稳定的环境，一直持续有新的feature/fix合并进来，QA也关注的环境
- 预发布环境，用于发布一个版本，在上线之间，进行完整验证用于给QA进行测试
- 生产环境，正确无误的代码，提供给用户使用



不同的环境应该完全隔离，比如db，cache之类的服务。
虽然一个db可以容纳多个表，但如果 db 宕机，所有的服务都将停止;
并且它们共享连接数，共享并发也是不合适的；
ci测试/内部测试会产生很多junk数据，到时也不容易清理；
